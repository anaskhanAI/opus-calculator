A

Use Case Complexity & Quantity

Pricing is directly linked to the number of use cases and their complexity. The first use case covers upfront fixed costs; each additional is priced at a reduced rate.

We categorize use cases by their Cognitive Load — how much "thinking" the automation must do.

"Walk me through the process — is it a fixed set of rules, or does it require decision-making based on data?"
Two Automation Tiers
Categorizing Use Case Complexity
Tier 1
Linear
Linear Automation (Deterministic)
Rule-based processes that follow a strict "Happy Path." Data is structured, outcome is predictable.
"When X happens, do Y" Form submissions Data syncing Simple notifications
Tier 2
Agentic
Agentic AI Automation (Probabilistic)
Processes that require "reasoning." Handles unstructured data, makes decisions, or involves Human-in-the-Loop approvals.
"Read this contract for risks" Email drafting from data Image analysis Decision-making flows
Pricing Logic
Volume-Based Pricing
💰 Pricing Principle

Base pricing: ~$80,000 for the first use case (covers fixed setup & delivery costs). Additional use cases: ~$20,000 each (up to 20 nodes, 5 integrations). As volume grows, per-use-case cost falls — maximizing customer ROI and accelerating adoption. Always use the Pricing Calculator for exact figures.

🔑 Multi-Use Case Selling Angle
  • As more use cases are added, the per-use-case cost becomes increasingly competitive
  • Frame the first use case as the foundation — subsequent ones are high-margin expansion
  • Emphasize broader deployment = deeper integration = expansion across the organization
  • Use the Delivery Pricing Calculator to generate exact customer-specific proposals

Integration Strategy

Integrations are the highest risk factor in delivery. Categorizing them correctly is critical to accurate pricing.

We charge based on engineering effort, not the name of the software. Our philosophy: Building vs. Configuring.

Q1: Which applications need to be connected?

Q2: How is data exchanged (REST/SOAP/DB/SDK)?

Q3: What level of documentation is available?
Four Integration Categories
What We Build vs. What We Configure
1CONFIG
Pre-Built Activation
You need a common tool (Gmail, Slack) and the required actions are already in our library. Configuring
  • Credential setup & secure linking
  • Connection testing & verification
  • Permissions audit
  • Workflow mapping & configuration
  • Immediate deployment (same day)
2EXTEND
Extended Integration
We're connected to your software, but your workflow requires an action we haven't mapped yet. Modifying
2A: Standard (REST) · 2B: Complex (Legacy/SOAP)
  • Endpoint identification & data mapping
  • Error handling logic
  • Unit testing of new action
  • 2B: XML/SOAP code structure & retry mechanisms
  • 2B: UAT in customer environment
3BUILD
New Standard Build
A modern SaaS tool we don't support yet. Good documentation, REST API. Building
  • Authentication build (OAuth/API Keys)
  • Base connector setup
  • Action development for required workflows
  • Rate limit management
  • Security review & encryption
4COMPLEX
New Complex Build
Older database, system behind a firewall, or specialized industry tool without modern APIs. Building
  • Network config (VPN/IP whitelisting)
  • Custom protocol agent on-server
  • Complex data parsing & serialization
  • Security compliance checks
  • Full-scale sandbox testing

Deployment Strategy

Deploying an enterprise platform is a controlled operational rollout — balancing speed, security, reliability, and commercial predictability.

Customers are not buying engineering hours. They are buying business outcomes: time-to-production certainty, reduced risk, compliance readiness, operational accountability.

Q1: Where should the solution be deployed?

Q2: Will AppliedAI have admin access?

Q3: Any security/compliance standards?

Q4: What network access model is required?
Six Deployment Tiers
Deployment Complexity Matrix
Tier Model Environment Delivery Complexity Key Characteristics
C1 Vendor Cloud (Multi-Tenant) Opus Cloud Low Fastest onboarding. Shared platform, logical isolation. Standardized infra, applicable for PoVs, not applicable for Enterprise deployments.
C2 Vendor Cloud (Dedicated Tenant) Opus Cloud Low–Medium Dedicated infra per customer. Higher customization and control.
C3 Customer Cloud (Vendor Access) AWS / Azure / GCP Medium Opus deploys in customer cloud with access. Standard automation + adjustments.
C4 Customer Cloud (No Vendor Access) AWS / Azure / GCP Medium–High "Blind Deployment" — customer teams execute using Opus IaC, Helm charts, runbooks. Slower troubleshooting.
C5 On-Prem / Private Cloud Customer Data Centre High Non-cloud-native. Custom integration, manual adjustments, extended validation.
C6 Air-Gapped / Restricted Isolated Network Very High Fully offline. Manual artifact distribution, strict security controls, extended timelines.

The Enterprise Delivery Journey
Three Phases to Go-Live
1
Readiness — Architecture & Environment Validation
  • Target architecture definition & Kubernetes readiness assessment
  • Identity, access & security validation. Compliance alignment (SOC2/HIPAA)
  • Risk identification, gap analysis & locked delivery plan
  • Outcome: Clear go/no-go decision. Deployment risk reduced before execution.
2
Platform Deployment & Environment Enablement
  • Foundation services + observability layer (logging, monitoring, certificates)
  • Core platform & workflow services deployment. Secure secrets configuration
  • For no-access deployments: IaC/Helm artifacts + guided execution
  • Outcome: Platform fully deployed. Environment-specific constraints respected.
3
Validation, Go-Live & Operational Handover
  • Staging: Full deployment, smoke testing, workflow validation, UAT
  • Production: Final deployment, configuration hardening, controlled go-live
  • Real-time monitoring during initial production. Post-go-live stabilization
  • Outcome: Production live. Runbooks & handover documentation delivered.
